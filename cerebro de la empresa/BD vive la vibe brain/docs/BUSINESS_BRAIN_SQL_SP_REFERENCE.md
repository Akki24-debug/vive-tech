# Business Brain Stored Procedures Reference
Ultima actualizacion: 2026-03-21
## Objetivo
Este documento es la referencia operativa principal para IA, backend y soporte al trabajar con los stored procedures de `vive_la_vibe_brain`.
Resume entradas, salidas, seguridad, convenciones y rutas recomendadas antes de invocar cualquier SP.
## Convenciones clave
- Lectura estandar: `sp_<entidad>_data`.
- Escritura base: `sp_<entidad>_upsert` o `sp_<entidad>_create`.
- Delete fisico solo en tablas puente o tecnicas autorizadas.
- Todos los writes usan `sp_actor_assert` y registran `audit_log`.
- Cuando cambia un campo de estado real, el write registra `status_history`.
- El bootstrap solo aplica a `organization`, `user_account`, `role` y `user_role`, y solo mientras no existan usuarios.
## Routing rapido para IA
- Si la intención es listar o buscar, usa primero `sp_<entidad>_data`.
- Si la intención es crear o editar una fila base, usa `sp_<entidad>_upsert`.
- Si la intención es registrar una nota o evento historico, usa los SP `*_create` append-only.
- Si la intención es cambiar estado operativo, prefiere el flujo compuesto (`sp_task_status_update`, `sp_project_status_update`, etc.) antes que tocar el `upsert` base.
- Si la intención es sincronizar relaciones many-to-many, usa `*_sync` en vez de llamar múltiples deletes/inserts sueltos.

## Notas de seguridad
- Las tablas globales o tecnicas sin `organization_id` directo validan actor, pero no siempre pueden filtrar org-scope por SQL directo.
- Las tablas append-only no exponen update/delete general aunque tengan `id`.
- Las tablas `audit_log` y `status_history` son de lectura para consumo directo; sus writes deben pasar por helpers.
